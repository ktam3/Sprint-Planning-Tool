# Sprint Planning Tool - Bug Fixes (March 18, 2026)

## Summary
Fixed 3 critical bugs that were causing incorrect velocity calculations and missing already-planned sprint items.

---

## Bug #1: Case-Sensitive Sprint State Check

**Problem:**
- Jira API returns sprint state as lowercase: `"future"`, `"active"`, `"closed"`
- Code was checking for uppercase: `"FUTURE"`, `"ACTIVE"`, `"CLOSED"`
- Result: Future sprints were being excluded, causing already-planned items to be missed

**Fix:**
- Made sprint state comparison case-insensitive using `.upper()`
- Updated in two locations:
  - Line 656: `if sprint_state.upper() not in ['ACTIVE', 'FUTURE']:`
  - Line 607: `if sprint_info and sprint_info.get('state', '').upper() in ['ACTIVE', 'FUTURE']:`

**Impact:**
- Llama Stack Core: Now detects 4 items in Sprint 14 (was only detecting 1)
- All teams: Future sprint items now properly detected

---

## Bug #2: Pagination Duplicates

**Problem:**
- Used offset-based pagination with `startAt` parameter
- Jira's `/rest/api/3/search/jql` endpoint uses cursor-based pagination with `nextPageToken`
- Result: Same 100 issues returned repeatedly, causing duplicates

**Fix:**
- Implemented cursor-based pagination in `search_issues()` method (lines 91-143)
- Use `nextPageToken` instead of `startAt`
- Check `isLast` flag to determine when to stop

**Code:**
```python
# Add pagination token if we have one
if next_page_token:
    params['nextPageToken'] = next_page_token

# Check if this is the last page
if data.get('isLast', True):
    break

# Get next page token
next_page_token = data.get('nextPageToken')
```

**Impact:**
- Now fetches all issues beyond the first 100
- Eliminates duplicate issue counting
- More accurate backlog and velocity data

## Bug #3: Velocity Inflation (CRITICAL)

**Problem:**
- Issues carried over across multiple sprints were counted in EVERY closed sprint
- Example: Issue in Sprints 10, 11, and 12 had its story points counted 3 times
- Result: Velocity inflated by 200-300%

**Before Fix Example:**
- Sprint 12: 174.5 pts (actually only 100 pts completed)
- Sprint 11: 129 pts (actually only 19 pts completed)
- Sprint 10: 84 pts (actually only 3 pts completed)
- **Calculated velocity: 129.2 pts/sprint**

**After Fix Example:**
- Sprint 12: 100 pts
- Sprint 11: 19 pts
- Sprint 10: 3 pts
- **Calculated velocity: 40.7 pts/sprint** ✅

**Fix:**
- Modified `calculate_velocity()` method (lines 245-295)
- Now only counts each issue in the MOST RECENT closed sprint it appears in
- Prevents double-counting of carried-over issues

**Code:**
```python
# Find the MOST RECENT closed sprint that matches the pattern
# Only count the issue in that sprint to avoid double-counting
latest_sprint_name = None
latest_sprint_date = None

for sprint_item in sprint_field:
    # ... check if sprint is closed and matches pattern
    if end_date >= cutoff_date:
        # Track the most recent sprint
        if latest_sprint_date is None or end_date > latest_sprint_date:
            latest_sprint_date = end_date
            latest_sprint_name = sprint_name

# Only count the issue in the most recent closed sprint
if latest_sprint_name:
    sprint_data_count[latest_sprint_name]['total'] += 1
    if status in done_statuses:
        sprint_data_count[latest_sprint_name]['completed'] += 1
```

**Impact:**
- Llama Stack Core: Velocity corrected from 129.2 to 95.8 pts/sprint
- Training Kubeflow: Velocity now accurate at 37.7 pts/sprint
- All teams: Realistic capacity planning
